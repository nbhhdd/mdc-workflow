# MDC 工作流方法论详解

## 一、为什么需要多文档协同？

### 1.1 大模型的本质：概率预测

大模型（LLM）本质上是一个基于参数权重的**概率预测引擎**。它不是"理解"你的项目，而是在做"下一个最可能的 token 是什么"的数学计算。

- 输入模糊 → 概率分布发散 → 幻觉（瞎编）
- 输入精确 → 概率分布收窄 → 高质量输出

**核心洞察**：AI 的输出质量不是由 AI 单方面决定的，而是由你的**约束精度**决定的。

### 1.2 单文档的瓶颈

传统做法是把所有需求、架构、规范都写在一个大文档里。这导致：

1. **上下文稀释**：几十页文档塞给 AI，它会"失忆"——注意力机制被无关信息摊薄
2. **修改耦合**：改一个小节可能影响全文结构，难以维护
3. **查找低效**：AI 无法快速定位"修这个 Bug 应该读哪一段"

### 1.3 多文档解耦的优势

把文档拆成独立模块，每次只喂给 AI 当前任务需要的：

```
修 Bug → 只读"错误档案" + "技术架构"
新功能 → 只读"功能规划" + "技术架构" + "代码规范"
重构 → 只读"审计方案" + "代码规范" + "文件规范"
```

这本质上是软件工程"高内聚、低耦合"原则在 prompt 工程中的映射。

## 二、TS-Protocol：时间戳闭环更新机制

### 2.1 为什么需要它？

在长周期人机协作中，最常见的故障模式是：

> 代码迭代了 N 版 → 文档停在 V1 → AI 拿着旧文档产生错误联想 → 输出偏离当前架构的代码

这是**人因故障**，不是 AI 故障。因为人忘记更新文档了。

### 2.2 TS-Protocol 如何解决

TS-Protocol 将"更新文档"从人类的待办事项转移为 AI 的自动化流程：

```
AI 改代码 → AI 查触发规则表 → AI 知道要更新哪些文档 → AI 更新文档 → AI 更新时间戳
```

核心价值：**从"人动"到"AI 动"**。

### 2.3 看门狗机制

时间戳同时充当"哨兵"：

```
AI 被要求修改代码
  → 先读关联文档的时间戳
  → 时间戳 vs 代码修改时间 对比
  → 如果文档过期：先同步文档，再写代码
  → 如果文档最新：直接开始工作
```

## 三、Router：任务-文档路由表

### 3.1 设计思想

Router 是一个轻量级的"编排器（Orchestrator）"。它不做复杂的逻辑推理，只做一件事：

> 给定任务类型 → 判断应该读哪些文档

### 3.2 为什么不用全自动

理论上可以让 AI 每次都通读所有文档。但实际上是：

- 大模型的注意力有限（即使有 100 万上下文窗口，注意力在长文本中也会稀释）
- 无关文档是"噪声"，会干扰 AI 对核心信息的提取
- 有选择地加载文档，等于给 AI 戴上了"聚焦镜"

### 3.3 路由规则设计原则

- **必读**：不读就会出错的文档（如修 Bug 必读错误档案）
- **可选**：读了有帮助但不读也能勉强工作的文档
- **不列**：当前任务完全无关的文档（如改 UI 不读测试策略）

## 四、Self-Reflection：AI 输出前自检清单

### 4.1 为什么要显式要求自反思

当代大模型虽然具备一定的推理能力，但在"抢答"模式下仍然容易：

- 忽略用户给定的约束条件
- 产生看似正确但违反架构原则的代码
- 引入不必要的复杂度

### 4.2 自检清单的心理学设计

11 条清单分为 5 组，故意控制了节奏：

| 组 | 条数 | 心理效果 |
|----|------|---------|
| 定位校验 | 2 条 | "先想清楚我在为谁做、做什么" |
| 架构校验 | 3 条 | "再检查骨架有没有问题" |
| 文档校验 | 2 条 | "确认文档同步责任" |
| 简洁性校验 | 2 条 | "收手，不要过度设计" |
| 安全校验 | 2 条 | "最后查底线问题" |

从"大方向"逐步收窄到"具体风险"，符合人类的审慎决策路径。

### 4.3 为什么 11 条而不是 3 条或 30 条

- 3 条太粗糙，无法覆盖关键风险面
- 30 条太长，AI 可能跳过或敷衍
- 11 条约需 30 秒即可完成一轮自检，成本极低

## 五、这套体系的边界与局限

### 适用场景

- 需要长期维护（≥3 个月）的软件项目
- 多人/多轮 AI 协作
- 技术栈和架构有一定复杂度的项目

### 不适用场景

- 一次性脚本或原型
- 几乎没有长期代码资产的小项目
- 完全由单人独立完成且不需要 AI 协作的项目

### 常见失败模式

1. **不坚持**：初期热情高，后面忘了更新文档 → 回到原点
2. **过度文档化**：为每个小函数写文档 → 文档膨胀、维护成本倒挂
3. **AI 跳过自检**：AI 有时会忽略自检清单 → 需要在 AGENTS.md 中反复强调
