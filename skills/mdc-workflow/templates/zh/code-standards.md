# 代码编写与 AI 协作规范

## 一、总体原则

代码必须服务长期维护。

优先级：

1. 清晰。
2. 稳定。
3. 可测试。
4. 可扩展。
5. 可读。

不追求：

1. 过早抽象。
2. 炫技写法。
3. 大而全重构。
4. 未验证的复杂架构。

## 二、知识库索引

本项目的完整文档索引、时间戳协议（TS-Protocol）、文档路由表见：

```text
project-docs/00-总览/README.md
```

每次协作前必须先读取上述入口文档，确认当前文档版本是否与代码同步。

## 四、参考规范

建议参考：

1. Tauri / Vue / React 等框架官方示例。
2. 各语言官方风格指南。
3. 项目内已有的代码模式。

## 五、AI 编程协作规则

每次改代码前：

1. 先读相关文件。
2. 先确认当前架构。
3. 不跨范围重构。
4. 不删除用户未确认的内容。
5. 先文档后大改。

每次改代码后：

1. 更新相关文档。
2. 执行构建检查。
3. 说明改动范围。
4. 说明未完成风险。

## 六、代码组织规则

### 前端

规则：

1. 一个组件只负责一个明确功能。
2. 大窗口用 `views`。
3. 复用 UI 用 `components`。
4. 状态放 `stores`。
5. 业务算法放 `engine` 或 `services`。
6. 框架调用封装到 `composables` 或专门 service。

### 后端

规则：

1. 命令/路由集中注册。
2. 不把业务逻辑全部塞进入口文件。
3. 按领域分模块。
4. 命令返回明确错误信息。

## 七、命名规范

1. 文件名使用 kebab-case 或框架约定。
2. 组件使用 PascalCase。
3. 函数名使用动词开头。
4. 类型名使用 PascalCase。
5. 常量使用 UPPER_SNAKE_CASE。

## 八、禁止事项

1. 禁止跨层级耦合（如后台功能塞进前台小窗口）。
2. 禁止新增无文档的大型模块。
3. 禁止硬编码大量业务数据而不说明来源。
4. 禁止把测试代码和正式代码混放。
5. 禁止提交临时修复脚本到根目录。

## 十一、AI 输出前自检清单 [Self-Reflection]

> 这是本项目 AI 协作的核心质量机制。AI 在输出任何代码方案前，必须逐条自问并确认通过。

### 定位校验

1. 这个改动是否符合"{{PROJECT_POSITIONING_SHORT}}"的产品定位？
2. 是否把不该放在当前层的功能塞进来了？（违反则重做）

### 架构校验

3. 是否引入了新的硬编码路径或魔法数字？（必须用配置/manifest 驱动）
4. 新增代码是否放对了目录？（查文件存放规范）
5. 是否遵循了"组件不承担算法、store 不管 IO、engine 不管 UI"的边界？

### 文档校验

6. 本次改动是否需要更新文档？（查 TS-Protocol 触发规则表）
7. 如果有文档需要更新，是否已经执行了文档更新时间戳？

### 简洁性校验

8. 这个方案有没有更简单的实现方式？是否引入了不必要的抽象或依赖？
9. 是否过度设计了"可能未来会用到"的功能？（YAGNI 原则）

### 安全校验

10. 敏感数据是否会被意外落盘或泄露？
11. 用户输入是否会被直接拼接到命令或查询中？

> 只有 11 条全部确认通过，才能开始输出代码。
